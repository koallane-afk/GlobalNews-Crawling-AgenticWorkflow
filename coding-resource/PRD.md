## 범위 선언

```
INPUT:  사용자가 지정한 뉴스 사이트 목록 (sources.yaml)
  ↓
[이 시스템이 하는 것]
  1. 모든 섹션을 동적으로 탐색하여 기사 수집
  2. 기사 제목 + 본문을 빅데이터 기법으로 분석
  3. 5-Layer 신호 분류 (유행 → 단기 → 중기 → 장기 → 싱귤래리티)
  ↓
OUTPUT: 구조화된 분석 데이터 (Parquet/SQLite)
        = "국내외 사회 변화 트렌드 연구의 기초 자료"
```

### 범위 밖 (명시적 제외)

다음은 이 시스템의 산출물(Parquet/SQLite)을 **입력으로 받는 별도 워크플로우**의 영역이다:

- 트렌드 리포트 자동 생성 (HTML/PDF)
- Streamlit/Dash 대시보드
- 실시간 알림/Push 시스템
- 시각화 차트/그래프 생성
- 협업/멀티유저 기능
- 소셜 미디어 통합
- SaaS 서비스화

## 1. 개요

### 1.1 문제 정의

사회과학 연구자, 데이터 저널리스트, 정책 분석가는 국내외 뉴스를 통해 사회 변화 트렌드를 파악해야 한다. 현재 이 작업은 다음과 같은 문제를 안고 있다:

1. **수동 수집의 비효율**: 연구자가 주 15-25시간을 뉴스 수집에 소비하며, 실제 분석 시간은 전체의 20-30%에 불과하다.
2. **표면적 수집**: 뉴스 사이트의 랜딩페이지 헤드라인만 수집하고, 각 섹션(정치/경제/사회/국제/IT/문화/스포츠/오피니언)의 심층 기사는 놓친다.
3. **분석 깊이 부족**: 수동으로는 토픽 모델링, 시계열 분석, 감정 분석, 네트워크 분석 등 빅데이터 기법을 적용할 수 없다.
4. **약한 신호 포착 불가**: 유행과 단기 트렌드는 감지하지만, 중기/장기 트렌드와 싱귤래리티(전례 없는 신호)는 놓친다.
5. **한국어+영어 이중 분석 불가**: 한국 뉴스와 글로벌 뉴스를 통합하여 교차 언어 분석을 수행할 수 없다.

### 1.2 솔루션

**GlobalNews Agentic Workflow**는 Claude Code(구독제)가 오케스트레이터 역할을 수행하며, 순수 Python 스크립트를 생성·실행하여 뉴스 크롤링과 빅데이터 분석을 **완전 자동화**하는 AI Agentic Workflow Automation System이다.

- **크롤링**: 사용자가 지정한 뉴스 사이트의 **모든 섹션**을 동적으로 탐색·수집한다. 차단 시 포기하지 않고 6-Tier 에스컬레이션으로 돌파한다.
- **분석**: 수집된 기사의 **제목과 본문**을 56개 빅데이터 분석 기법으로 처리하여, 유행/단기/중기/장기/싱귤래리티 5계층 신호를 자동 분류한다.
- **산출물**: 구조화된 분석 데이터(Parquet/SQLite)를 생산한다. 이것이 "국내외 사회 변화 트렌드 연구의 기초 자료"이다.

### 1.3 차별화 (3대 고유 가치)

| # | 차별화 | 설명 |
|---|--------|------|
| **D1** | **Dynamic-First 전체 사이트 크롤링** | 랜딩페이지가 아닌 모든 섹션을 Playwright/Patchright로 클릭 탐색. 기존 도구(GDELT, MediaCloud)가 RSS/API에 의존하는 것과 근본적으로 다르다. |
| **D2** | **적응형 차단 돌파 (Never Give Up)** | 7가지 차단 유형 실시간 진단 → 6-Tier 에스컬레이션 → Claude Code가 Python 우회 코드를 실시간 생성 → 사이트별 전략 학습. "Agentic"의 핵심. |
| **D3** | **한국어+글로벌 융합 분석** | Kiwi 형태소 분석 + KoBERT/KcELECTRA + 다국어 임베딩으로 한국 뉴스와 글로벌 뉴스를 동일 파이프라인에서 처리. |

## 2. 자동화 목적

### 2.1 현재 상태 (As-Is)

| 작업 | 현재 방식 | 소요 시간 | 문제 |
|------|----------|----------|------|
| 뉴스 수집 | 수동 브라우징 + RSS 리더 | 주 15-25시간 | 랜딩페이지만, 섹션 심층 기사 누락 |
| 기사 정리 | 스프레드시트 수동 입력 | 주 5-10시간 | 비구조화, 중복, 누락 빈번 |
| 기본 분석 | 키워드 검색, 수동 분류 | 주 5-10시간 | 단순 빈도 수준, 토픽 모델링 불가 |
| 트렌드 파악 | 직관적 판단 | 분석 시간의 일부 | 유행만 감지, 약한 신호 놓침 |
| **총계** | **수동** | **주 25-45시간** | **분석보다 수집에 80% 소비** |

### 2.2 목표 상태 (To-Be)

| 작업 | 자동화 방식 | 소요 시간 | 개선 |
|------|-----------|----------|------|
| 뉴스 수집 | Dynamic-First 크롤링 엔진 | 30-45분/일 (자동) | 모든 섹션 탐색, 인간 개입 불필요 |
| 기사 정리 | 자동 추출 + 중복 제거 + 구조화 | 실시간 (파이프라인) | Parquet 구조화, 0% 중복 |
| 빅데이터 분석 | 56개 기법 자동 파이프라인 | 15-30분/1,000기사 | NLP, 시계열, 네트워크, 감정 등 |
| 신호 분류 | 5-Layer 규칙 기반 자동 분류 | 실시간 (파이프라인) | 유행~싱귤래리티 전 계층 탐지 |
| **총계** | **완전 자동** | **일 1-2시간 (크롤링+분석)** | **연구자는 분석 결과 해석에 집중** |

### 2.3 자동화 범위

**자동화 대상 (이 시스템)**:

1. 뉴스 사이트 전체 섹션 탐색 및 URL 발견
2. 기사 본문 추출 및 구조화
3. 크롤링 차단 감지 및 자동 돌파
4. 텍스트 전처리 (한국어/영어)
5. 56개 빅데이터 분석 기법 실행
6. 5-Layer 신호 분류 및 태깅
7. 분석 결과를 Parquet/SQLite로 저장
8. 크롤링 스케줄링 (cron 기반)

**자동화 비대상 (별도 워크플로우)**:

- 분석 결과 해석 및 리포트 작성
- 시각화 대시보드 구축
- 사용자 인터페이스 (웹/앱)
- 알림/노티피케이션 시스템

### 2.4 Build vs Buy 판단

| 옵션 | 비용 | 적합성 | 판단 |
|------|------|--------|------|
| **GDELT** | 무료 | RSS/API 기반, 한국어 약함, 원문 미제공 | ❌ 전체 섹션 크롤링 불가 |
| **MediaCloud** | 무료 | 아카데믹, 한국 소스 부족, 분석 제한적 | ❌ 한국어 NLP 미지원 |
| **Event Registry** | $500+/월 | API 기반, 커스텀 분석 불가 | ❌ 예산 초과, 유연성 부족 |
| **Diffbot** | $300+/월 | 크롤링 강점, 분석 약함 | ❌ 빅데이터 분석 부재 |
| **Brandwatch/Meltwater** | $1,000+/월 | 소셜 미디어 중심, 뉴스 크롤링 제한 | ❌ 범위 불일치 |
| **자체 구축 (Build)** | $0-150/월 | 전체 커스터마이징, 56개 기법, 한국어 특화 | ✅ **채택** |

**Build 판단 근거**:
1. 기존 도구는 "전체 섹션 동적 크롤링"을 지원하지 않음 (RSS/API 의존)
2. 56개 빅데이터 분석 기법을 커스텀 조합할 수 있는 상용 도구 부재
3. 한국어 NLP(Kiwi/KoBERT)와 영어 NLP를 단일 파이프라인에 통합하는 도구 부재
4. Claude Code 구독제로 오케스트레이션 비용 $0 달성 가능
5. 월 변동비 $0-150(프록시만)으로 상용 도구 대비 1/10 이하

### 4.3 3-Tier URL 발견 전략

뉴스 사이트에서 기사 URL을 발견하는 3단계 접근:

| Tier | 방법 | 커버리지 | 차단 위험 | 설명 |
|------|------|---------|----------|------|
| **Tier 1** | Sitemap + RSS | 60-70% | 없음 | `sitemap.xml`, `rss.xml` 파싱. 차단 위험 0. |
| **Tier 2** | DOM 네비게이션 | 추가 15-20% | 낮음 | HTML 네비게이션 메뉴에서 섹션 URL 추출, 각 섹션 페이지 크롤링 |
| **Tier 3** | Playwright 동적 크롤링 | 나머지 10-20% | 중간 | 무한스크롤, Load More 버튼, JavaScript 렌더링이 필요한 콘텐츠 |

**탐색 순서**: Tier 1 → Tier 2 → Tier 3. 낮은 비용/위험부터 시작하여 점진적으로 깊이를 더한다.

## 5. 핵심 기능

### 5.1 크롤링 엔진

#### 5.1.1 Dynamic-First 전체 사이트 크롤링

**원칙**: 랜딩페이지 헤드라인만이 아닌, 뉴스 사이트의 **모든 섹션**을 동적으로 탐색한다.

**탐색 방식**:

```
사이트 진입
  ↓
[Phase 1: 정적 발견]
  • sitemap.xml 파싱 → 전체 URL 목록 확보 (60-70%)
  • RSS 피드 파싱 → 최신 기사 URL 확보
  ↓
[Phase 2: DOM 탐색]
  • 네비게이션 메뉴에서 섹션 URL 추출
  • 각 섹션 페이지 방문 → 기사 링크 추출
  • 페이지네이션 처리 (다음 페이지 → 다음 페이지 → ...)
  ↓
[Phase 3: 동적 크롤링]
  • Playwright/Patchright로 JavaScript 렌더링
  • 무한 스크롤 처리 (scroll → wait → scroll)
  • "Load More" 버튼 자동 클릭
  • AJAX로 로딩되는 기사 목록 처리
  ↓
[Phase 4: 엣지 케이스]
  • 실패한 URL 재시도 (exponential backoff)
  • 실패 로그 → Claude Code 대화형 분석 (Tier 6)
```

**섹션 탐색 목록** (사이트별 자동 매핑):

- 정치, 경제, 사회, 국제, IT/과학, 문화, 스포츠, 오피니언/칼럼
- 각 섹션의 서브섹션 (예: 경제 → 증권, 부동산, 산업)
- 특집/기획 섹션

**기사 추출 필드**:

| 필드 | 추출 방법 | 필수 여부 |
|------|----------|---------|
| `url` | 크롤링 URL | 필수 |
| `title` | `<h1>`, `og:title`, schema.org | 필수 |
| `body` | Trafilatura/Fundus 본문 추출 | 필수 |
| `published_at` | `article:published_time`, schema.org | 필수 |
| `author` | `article:author`, byline | 선택 |
| `category` | URL 경로, 메타데이터 | 선택 |
| `source` | 소스 사이트명 | 필수 |

#### 5.1.2 적응형 차단 돌파 시스템 (Adaptive Anti-Block Resilience)

**원칙**: 크롤링이 차단되더라도 **포기하지 않는다**. 차단 유형을 실시간 진단하고, 단계적으로 에스컬레이션하여 돌파한다.

**7가지 차단 유형 진단**:

| # | 차단 유형 | 증상 | 자동 진단 |
|---|----------|------|----------|
| B1 | **IP 차단** | HTTP 403/429 | 응답 코드 모니터링 |
| B2 | **User-Agent 필터링** | 특정 UA 차단 | UA 회전 후 응답 비교 |
| B3 | **Rate Limiting** | 연속 요청 시 지연/차단 | 요청 간격 vs 응답 시간 상관 |
| B4 | **CAPTCHA** | CAPTCHA 페이지 리다이렉션 | HTML 패턴 감지 |
| B5 | **JavaScript 챌린지** | Cloudflare/Akamai 챌린지 | JS 실행 요구 감지 |
| B6 | **핑거프린트 탐지** | 브라우저 특성 기반 차단 | CDP 프로토콜 감지 |
| B7 | **Geo-Block** | IP 지역 기반 차단 | 다른 지역 프록시로 확인 |

**6-Tier 에스컬레이션**:

| Tier | 전략 | 비용 | 성공률 | 자동화 |
|------|------|------|--------|--------|
| **Tier 1** | 요청 간격 조정 (5초→10초→15초) + User-Agent 회전 | $0 | 높음 | 완전 자동 |
| **Tier 2** | 세션 관리 (쿠키 순환 + Referer 설정) + HTTP 헤더 다변화 | $0 | 높음 | 완전 자동 |
| **Tier 3** | Playwright/Patchright 동적 렌더링 (headless 브라우저) | $0 | 중-높 | 완전 자동 |
| **Tier 4** | Patchright + Apify fingerprint-suite (핑거프린트 위장) | $0 | 중간 | 완전 자동 |
| **Tier 5** | 주거용 프록시 회전 (DataImpulse 등) | $0.10-1/GB | 중-높 | 완전 자동 |
| **Tier 6** | Claude Code 대화형 분석 — 실패 로그 리뷰 + 신규 Python 우회 코드 생성 | $0 (구독 내) | 가변 | **반자동** |

**Tier 6 상세 — Claude Code의 역할**:

Claude Code(구독제)는 일상적 크롤링에는 관여하지 않는다. Tier 1-5가 모두 실패한 엣지 케이스에서만 활성화된다:

1. 실패 로그 파일을 읽는다 (HTTP 응답 코드, 에러 메시지, 차단 패턴)
2. 차단 유형을 분석한다
3. 해당 사이트에 특화된 Python 우회 스크립트를 생성한다
4. Bash로 실행하여 결과를 확인한다
5. 성공 시 전략을 사이트별 프로파일에 저장하여 재사용한다

**Circuit Breaker 패턴**:

| 상태 | 조건 | 동작 |
|------|------|------|
| **Closed** (정상) | 연속 성공 | 정상 크롤링 계속 |
| **Open** (차단) | 연속 실패 5회 | 크롤링 중단, 30분 대기, Tier 에스컬레이션 |
| **Half-Open** (시험) | 대기 후 | 단일 요청으로 테스트, 성공 시 Closed |

| **교차 차원 분석** | 4 | 교차 국가 비교, 프레임 분석, 의제 설정, 시간 정렬 | sentence-transformers, scipy |
| **AI/ML 분석** | 6 | Zero-shot 분류, 인과 추론(PCMCI), 서사 추출, 모순 탐지, SetFit 분류, GraphRAG | setfit, tigramite, transformers(로컬) |

> **중요**: "AI/ML 분석" 영역의 모든 기법은 **로컬에서 실행되는 오픈소스 모델**을 사용한다. Claude API를 호출하지 않는다. Zero-shot 분류는 `facebook/bart-large-mnli` 등 로컬 모델을, SetFit는 8개 예시만으로 fine-tuning하는 few-shot 모델을 사용한다.

#### 5.2.2 8-Stage 분석 파이프라인

```
Stage 1: 전처리
  • 한국어: Kiwi 형태소 분석 → 명사/동사/형용사 추출 → 불용어 제거
  • 영어: spaCy → lemmatization → 불용어 제거
  • 공통: 문장 분리, 정규화, 언어 감지
  ↓
Stage 2: 특성 추출
  • SBERT 임베딩 (한국어: snowflake-arctic-embed-l-v2.0-ko)
  • TF-IDF (단어/바이그램)
  • NER (한국어: KLUE-RoBERTa-large, 영어: spaCy)
  • KeyBERT 키워드 추출
  • FastText 단어 벡터 (언어별)
  ↓
Stage 3: 기사별 분석
  • 감정 분석 (한국어: KoBERT F1=94%, 영어: 로컬 transformer)
  • 8차원 감정 분류 (Plutchik 모델: 기쁨/신뢰/공포/놀람/슬픔/혐오/분노/기대)
  • Zero-shot 주제 분류 (STEEPS: Social/Technology/Economic/Environmental/Political/Security)
  • 중요도 점수 산출
  ↓
Stage 4: 집계 분석
  • BERTopic 토픽 모델링 (Model2Vec로 CPU 500x 속도)
  • Dynamic Topic Modeling (시간별 토픽 변화)
  • HDBSCAN 클러스터링
  • NMF/LDA 보조 토픽
  • 커뮤니티 탐지 (Louvain)
  ↓
Stage 5: 시계열 분석
  • STL 분해 (트렌드/계절/잔차)
  • Kleinberg 버스트 탐지 (이슈 폭발)
  • 변화점 감지 — PELT 알고리즘 (ruptures)
  • Prophet 예측 (다음 7/30일)
  • Wavelet 분석 (다중 주기)
  • 이동평균 교차 (단기/장기)
  ↓
Stage 6: 교차 분석
  • Granger 인과 검정 (토픽 간 시간적 선행 관계)
  • PCMCI 인과 추론 (tigramite — 다변량 시계열)
  • 공출현 네트워크 (entity-entity, topic-topic)
  • 교차 언어 토픽 정렬 (한국어↔영어)
  • 프레임 분석 (동일 이슈의 보도 프레임 비교)
  ↓
Stage 7: 신호 분류 (5-Layer)
  • 규칙 기반 5-Layer 분류 (아래 상세)
  • Novelty 탐지 (LOF, Isolation Forest)
  • BERTrend weak signal → emerging signal 전환 감지
  • 싱귤래리티 복합 점수 산출
  ↓
Stage 8: 데이터 출력
  • Parquet (ZSTD 압축) — 분석 결과 데이터
  • SQLite (FTS5 전문 검색 + sqlite-vec 벡터 검색) — 인덱스/쿼리
  • DuckDB 쿼리 호환
```

#### 5.2.3 5-Layer 신호 탐지 체계

| Layer | 신호 유형 | 시간 범위 | 핵심 탐지 기법 | 최소 데이터 |
|-------|----------|----------|-------------|-----------|
| **L1** | **유행 (Fad)** | 일 ~ 2주 | Kleinberg 버스트 + Z-score 이상 탐지 | 7일 |
| **L2** | **단기 트렌드** | 2주 ~ 3개월 | BERTopic DTM + 감정 궤적 + 이동평균 교차 | 30일 |
| **L3** | **중기 트렌드** | 3개월 ~ 1년 | 변화점 감지(PELT) + 네트워크 진화 + 프레임 분석 | 6개월 |
| **L4** | **장기 트렌드** | 1 ~ 5년 | 임베딩 드리프트 + Wavelet + STEEPS 분류 | 2년+ |
| **L5** | **싱귤래리티** | 비주기 | Novelty 점수 + 교차 도메인 출현 + BERTrend weak→emerging 전환 + 복합 7지표 점수 | 6개월+ |

**Dual-Pass 분석 전략 (K3 준수)**:

| Pass | 대상 | 역할 | 기법 |
|------|------|------|------|
| **Pass 1: 제목** | 기사 제목 | 신호 탐지 (빠른 스캔) | 키워드 빈도, 버스트 감지, 감정 극성 |
| **Pass 2: 본문** | 기사 전문 | 증거 확보 (심층 분석) | NER, 토픽 모델링, 프레임 분석, 인과 추론 |

> 제목은 "무엇이 일어나고 있는가"를, 본문은 "왜, 어떻게 일어나고 있는가"를 답한다.

#### 5.2.4 싱귤래리티 탐지 (L5 상세)

"전례 없는 신호"를 탐지하기 위한 3가지 독립 경로 교차 검증:

| 경로 | 방법 | 설명 |
|------|------|------|
| **OOD 감지** | Out-of-Distribution (LOF, Isolation Forest) | 기존 토픽 분류에 속하지 않는 기사 감지 |
| **변화점 감지** | PELT + 확률 분포 변화 | 기사 빈도, 감정, 토픽 분포의 구조적 전환점 |
| **BERTrend** | weak signal → emerging 전환 | 약한 신호가 처음 떠오르는 시점 포착 |

**싱귤래리티 복합 점수 공식**:

```
S_singularity = w1 × OOD_score
              + w2 × Changepoint_significance
              + w3 × CrossDomain_emergence
              + w4 × BERTrend_transition
              + w5 × Entropy_spike
              + w6 × Novelty_score
              + w7 × Network_anomaly

(w1-w7: 도메인 전문가 또는 그리드 서치로 조정)
```

## 6. 시스템 아키텍처

### 6.1 Staged Monolith 아키텍처

마이크로서비스가 아닌 **단계적 모놀리스(Staged Monolith)** 접근을 채택한다.

**근거**:
- 단일 머신(MacBook)에서 실행
- 컴포넌트 간 통신 오버헤드 불필요
- 배포/운영 복잡성 최소화
- 필요 시 각 Stage를 독립 프로세스로 분리 가능 (모듈 경계 유지)

```
┌──────────────────────────────────────────────────────────────────┐
│                     GlobalNews Agentic Workflow                    │
│                                                                    │
│  ┌─────────────────────────────────────────────────────────┐      │
│  │              Orchestration Layer                          │      │
│  │  ┌────────────┐   ┌─────────────┐   ┌──────────────┐   │      │
│  │  │ cron       │   │ Claude Code │   │ 상태 관리    │   │      │
│  │  │ (스케줄링) │   │ (구독제     │   │ (state.yaml) │   │      │
│  │  │            │   │  엣지 전용) │   │              │   │      │
│  │  └────────────┘   └─────────────┘   └──────────────┘   │      │
│  └─────────────────────────────────────────────────────────┘      │
│                              ↓                                     │
│  ┌─────────────────────────────────────────────────────────┐      │
│  │              Crawling Layer (Python)                      │      │
│  │  RSS/Sitemap → HTTP/Trafilatura → Playwright/Patchright  │      │
│  │  → 차단 진단 → 6-Tier 에스컬레이션 → Circuit Breaker    │      │
│  └─────────────────────────────────────────────────────────┘      │
│                              ↓                                     │
│  ┌─────────────────────────────────────────────────────────┐      │
│  │              Analysis Layer (Python — 8 Stages)           │      │
│  │  전처리 → 특성추출 → 기사별분석 → 집계 → 시계열          │      │
│  │  → 교차분석 → 신호분류 → 데이터출력                       │      │
│  └─────────────────────────────────────────────────────────┘      │
│                              ↓                                     │
│  ┌─────────────────────────────────────────────────────────┐      │
│  │              Storage Layer                                │      │
│  │  Parquet (ZSTD) │ SQLite (FTS5+vec) │ DuckDB (쿼리)     │      │
│  └─────────────────────────────────────────────────────────┘      │
└──────────────────────────────────────────────────────────────────┘
```

### 6.4 Claude Code 오케스트레이터 패턴 (Conductor Pattern)

Claude Code(구독제)는 데이터를 직접 처리하지 않는다. **Generate → Execute → Read → Decide** 루프를 수행한다:

```
┌──────────────────────────────────────────────┐
│           Claude Code (구독제)                 │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐   │
│  │ Generate  │→│ Execute   │→│ Read &    │   │
│  │ Python    │  │ via Bash  │  │ Decide    │   │
│  │ Scripts   │  │ Tool      │  │ Next Step │   │
│  └──────────┘  └──────────┘  └──────────┘   │
│       ↑                            │          │
│       └────────────────────────────┘          │
│              (결과 기반 다음 판단)              │
└──────────────────────────────────────────────┘
```

**역할 분리**:

| 컴포넌트 | 역할 | 비용 |
|---------|------|------|
| **cron + Python** | 일상 크롤링 + 분석 (95%) | $0 |
| **Claude Code** | 워크플로우 설계, 스크립트 생성, 엣지 케이스 해결 (5%) | $0 (구독 내) |

### 기술 용어

| 용어 | 정의 |
|------|------|
| **Agentic Workflow** | AI 에이전트가 자율적으로 판단하며 작업을 수행하는 자동화 워크플로우 |
| **Dynamic Crawling** | JavaScript 렌더링이 필요한 웹페이지를 브라우저 자동화로 수집하는 기법 |
| **BERTopic** | BERT 임베딩 + HDBSCAN 클러스터링 + TF-IDF 기반 토픽 모델링 기법 |
| **Patchright** | Playwright의 CDP(Chrome DevTools Protocol) 탐지를 우회하는 패치 버전 |
| **Parquet** | Apache Arrow 기반 열지향 데이터 포맷. 효율적 압축과 빠른 쿼리 |
| **FTS5** | SQLite의 Full-Text Search 5 엔진. 전문 검색 지원 |
| **sqlite-vec** | SQLite의 벡터 유사도 검색 확장 |
| **SBERT** | Sentence-BERT. 문장 수준 임베딩을 생성하는 Transformer 모델 |
| **Circuit Breaker** | 연속 실패 시 자동으로 요청을 중단하고 복구를 시도하는 패턴 |
| **Conductor Pattern** | 오케스트레이터가 스크립트를 생성·실행하고 결과를 해석하는 패턴 |

| 약어 | 풀네임 |
|------|--------|
| **NER** | Named Entity Recognition (개체명 인식) |

### 도메인 용어

| 용어 | 정의 |
|------|------|
| **싱귤래리티 신호** | 기존 카테고리에 속하지 않는, 전례 없는 사회 변화 신호 |
| **5-Layer 신호 체계** | 유행/단기/중기/장기/싱귤래리티로 트렌드를 시간 범위별 분류하는 체계 |

| **Kleinberg 버스트** | 기사 빈도의 급격한 증가를 통계적으로 감지하는 알고리즘 |
| **사회 무드 지수** | 특정 기간의 뉴스 감정 분포를 종합한 사회 분위기 지표 |
